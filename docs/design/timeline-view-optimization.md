# OpenClaw Agent Dashboard 时序视图优化方案

## 一、现状问题

| 问题 | 表现 |
|------|------|
| 语义不清晰 | 「调用」「结果」等标签过于抽象，难以理解 Agent 在做什么 |
| LLM 边界不明确 | 无法直观看出哪些步骤是 LLM 输出、哪些是工具执行 |
| 工具链路不直观 | toolCall 与 toolResult 的对应关系不清晰 |
| 失败原因难定位 | 「结果 失败」缺少工具名、错误类型等上下文 |
| 信息密度低 | 折叠后信息过少，展开又显得冗长 |

## 二、优化目标

- 一眼能看懂：用户能快速理解每一步在做什么、谁在做
- 突出 LLM 调用：明确区分「LLM 输出」与「工具执行」
- 强化因果链：toolCall ↔ toolResult 成对展示
- 失败可解释：失败时能快速看到工具名、错误类型和简要建议

## 三、优化方案

### 3.1 步骤标签与文案优化

| 当前 | 优化后 | 说明 |
|------|--------|------|
| 调用 | Read 调用 / Bash 调用 | 用「工具名 + 调用」替代单一「调用」 |
| 结果 | Read 结果 / Read 失败 | 用「工具名 + 结果/失败」 |
| 思考 | 思考（副标题：LLM 推理中） | 明确是 LLM 推理过程 |
| 响应 | 回复（副标题：LLM 输出） | 明确是 LLM 文本输出 |
| 用户 | 保持或改为 用户输入 / 子 Agent 回传 | 区分真实用户与子 Agent 回传 |

### 3.2 视觉分组：按「LLM 轮次」分组

**LLM 轮次 #1**

```
+-- LLM 轮次 #1 -----------------------------------+
|  🧠 思考  用户需要分析项目，先读取目录...         |
|  🔧 Read 调用  path: src/main.py                  |
+--------------------------------------------------+
```

**工具执行**

```
+-- 工具执行 ---------------------------------------+
|  ✅ Read 结果  成功返回 342 行                    |
+--------------------------------------------------+
```

**LLM 轮次 #2（处理 Read 结果）**

```
+-- LLM 轮次 #2（处理 Read 结果）------------------+
|  🧠 思考  已读取 main.py，继续分析...             |
|  🤖 回复  项目入口在 main.py，包含 FastAPI...     |
+--------------------------------------------------+
```

- 同一 assistant 消息内的 thinking + toolCall + text 归为同一「LLM 轮次」
- toolResult 单独作为「工具执行」块
- 用分组框或背景色区分「LLM 输出」与「工具执行」

### 3.3 工具调用与结果成对展示

- **连线**：toolCall 与对应 toolResult 之间用虚线或连接线
- **缩进**：toolResult 相对 toolCall 缩进，表示从属关系
- **折叠联动**：折叠 toolCall 时，可选项：同时折叠对应 toolResult，或保留 toolResult 摘要

### 3.4 折叠态摘要优化

| 步骤类型 | 当前折叠态 | 优化后折叠态 |
|---------|------------|--------------|
| toolCall | 🔧 Read | 🔧 Read src/main.py |
| toolResult 成功 | ✅ 结果 | ✅ Read 成功 · 342 行 |
| toolResult 失败 | ❌ 失败 | ❌ Read 失败 · 文件不存在 |
| thinking | 🧠 思考 | 🧠 思考 · 约 150 字 |
| text | 🤖 响应 | 🤖 回复 · 约 80 字 |

从「类型名」升级为「类型 + 关键信息」，便于快速扫读。

### 3.5 图例与说明

在时序区域顶部增加图例：

```
图例：🧠 LLM 输出  |  🔧 工具调用  |  ✅ 工具成功  |  ❌ 工具失败  |  👤 用户/回传
```

可选：增加「LLM 调用」与「工具执行」的简要说明，帮助新用户理解流程。

### 3.6 失败结果增强

对「结果 失败」增加结构化展示：

```
+-- Read 失败 -------------------------------------------+
|                                                         |
|  ❌ 工具执行失败                                        |
|                                                         |
|  错误：                                                 |
|  ENOENT: no such file or directory 'src/xxx.py'        |
|                                                         |
|  💡 建议：                                              |
|  • 检查文件路径是否存在                                 |
|  • 确认是否有读取权限                                   |
|                                                         |
+---------------------------------------------------------+
```

- 展示工具名、错误信息、可选错误类型
- 对常见错误（如 ENOENT、权限、超时）给出简短建议

### 3.7 可选：LLM 调用标记

在 LLM 输出步骤旁增加小标签，例如：

- `[LLM]` 或 `大模型输出`
- 或仅在首次出现时展示，后续用统一图标表示

## 四、实现优先级建议

| 优先级 | 项目 | 工作量 | 收益 |
|--------|------|--------|------|
| P0 | 步骤标签优化（工具名 + 调用/结果/失败） | 小 | 高 |
| P0 | 失败结果增强（错误信息 + 建议） | 小 | 高 |
| P1 | 折叠态摘要优化 | 中 | 高 |
| P1 | 顶部图例 | 小 | 中 |
| P2 | LLM 轮次分组 | 中 | 高 |
| P2 | toolCall ↔ toolResult 连线/缩进 | 中 | 中 |
| P3 | 子 Agent 回传单独样式 | 小 | 中 |

```

    │
    ▼

    │
    ▼

    │
    ▼
```

## 七、小结

核心思路：

- **语义更清晰**：用「工具名 + 调用/结果/失败」替代抽象标签
- **结构更清楚**：按 LLM 轮次分组，区分 LLM 输出与工具执行
- **失败更可解释**：展示错误信息和简要建议
- **折叠更有用**：折叠态展示关键信息，而不是仅类型名

建议先做 P0（标签 + 失败增强），再逐步推进 P1、P2。
