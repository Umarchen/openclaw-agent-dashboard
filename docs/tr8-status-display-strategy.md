# Agent 状态显示策略分析

## 问题本质

Agent 的思考、工具执行等动作通常非常快（几百毫秒到几秒），而文件读取存在不可避免的延迟：
- JSONL 写入延迟
- 文件系统 I/O 延迟
- 轮询/推送间隔延迟

**结论**：从技术上说，很难精确区分"正在思考"和"正在执行工具"，因为等读取到文件时，动作可能已经结束了。

---

## 用户真正关心的是什么？

### 不太有价值的显示

| 状态 | 问题 |
|------|------|
| "思考中..." | 持续时间短（通常 <1s），无法准确判断 |
| "执行: Read" | 如果只执行 50ms，显示出来已经过时 |
| "执行: Grep" | 同上，快速工具执行无意义 |

### 真正有价值的场景

**用户关心的核心问题是：卡住了吗？卡在哪里？**

| 场景 | 持续时间 | 用户关注度 |
|------|----------|------------|
| 等待模型响应 | 5s ~ 60s+ | 高 - 模型慢/网络问题/限流 |
| 等待子代理完成 | 10s ~ 数分钟 | 高 - 子代理任务进度 |
| 执行耗时工具（Bash 长命令） | 3s ~ 数分钟 | 高 - 命令卡住/资源阻塞 |
| 大文件读写 | 2s ~ 30s | 中 - 文件操作进度 |
| 正常快速执行 | <2s | 低 - 不需要显示 |

---

## 重新设计：基于"卡顿"的状态显示

### 核心思路

**只显示"有意义"的状态 = 超过阈值才显示具体动作**

```
                    时间轴
    ─────────────────────────────────────────────►

    工具开始      阈值(2s)       工具结束
       │            │              │
       ▼            ▼              ▼
    ┌──────┐    ┌──────┐      ┌──────┐
    │ 不显示 │───►│ 显示  │─────►│ 完成 │
    │       │    │ 具体工具│      │      │
    └──────┘    └──────┘      └──────┘
```

### 建议的状态体系

```

│  │ 工作中      │  任务刚开始，或快速执行中                       │

│  └─────────────┘                                                │
```

### 具体阈值建议

| 状态类型 | 阈值 | 显示内容 | 判断依据 |
|----------|------|----------|----------|
| 等待模型响应 | 5s | "等待响应" | 最后一条 assistant 消息后超过 5s 无新消息 |
| 等待模型（可能限流） | 15s | "等待响应 (可能限流)" | 同上，但超过 15s |
| 等待子代理 | 3s | "等待 [agent名]" | 有未完成的子代理 run |
| Bash 命令执行 | 2s | "执行命令" | toolCall=Bash 且无 toolResult |
| 大文件操作 | 2s | "读写文件" | toolCall=Write/Edit 且无 toolResult |
| 其他工具 | 3s | "执行工具" | 其他 toolCall 且无 toolResult |
| 快速执行中 | - | "处理中..." | 有任务但未超过任何阈值 |

## 实现方案

### 状态判断逻辑（伪代码）

```python
def get_display_status(agent_id: str) -> dict:
    """
    返回用于前端显示的状态

    Returns:
        {
            'status': 'idle' | 'working',
            'display': str,  # 显示文本
            'detail': str | None,  # 详细信息（悬停显示）
            'duration': int,  # 持续时间（秒）
            'alert': bool,  # 是否需要警告（如限流）
        }
    """
    # 检查是否有活跃任务
    if not is_agent_working(agent_id):
        return {'status': 'idle', 'display': '空闲'}

    # 计算空闲时间（距离最后一次活动）
    idle_seconds = get_idle_seconds(agent_id)

    # 检查是否在等待子代理
    waiting_for = get_waiting_child_agent(agent_id)
    if waiting_for:
        if idle_seconds > 3:
            return {
                'status': 'working',
                'display': f'等待 {waiting_for}',
                'detail': f'子代理任务进行中，已等待 {idle_seconds}s',
                'duration': idle_seconds,
                'alert': idle_seconds > 60  # 超过1分钟提示
            }
        else:
            return {'status': 'working', 'display': '处理中...'}

    # 检查是否有未完成的工具调用
    tool_call = get_pending_tool_call(agent_id)
    if tool_call:
        tool_name = tool_call.get('name', '')
        tool_duration = get_tool_duration(tool_call)

        # 根据工具类型和持续时间决定显示
        if tool_name == 'Bash' and tool_duration > 2:
            return {
                'status': 'working',
                'display': '执行命令',
                'detail': f'Bash 命令执行中，已耗时 {tool_duration}s',
                'duration': tool_duration,
                'alert': tool_duration > 30
            }
        elif tool_name in ('Write', 'Edit') and tool_duration > 2:
            return {
                'status': 'working',
                'display': '读写文件',
                'detail': f'{tool_name} 执行中，已耗时 {tool_duration}s',
                'duration': tool_duration,
                'alert': False
            }
        elif tool_duration > 3:
            return {
                'status': 'working',
                'display': '执行工具',
                'detail': f'{tool_name} 执行中，已耗时 {tool_duration}s',
                'duration': tool_duration,
                'alert': False
            }
        else:
            return {'status': 'working', 'display': '处理中...'}

    # 没有待处理工具，检查是否在等待模型
    if idle_seconds > 5:
        alert = idle_seconds > 15  # 可能限流
        return {
            'status': 'working',
            'display': '等待响应' + (' (可能限流)' if alert else ''),
            'detail': f'等待模型响应，已等待 {idle_seconds}s',
            'duration': idle_seconds,
            'alert': alert
        }

    # 刚开始或快速执行中
    return {'status': 'working', 'display': '处理中...'}
```

## 关键改进点

### 1. 去掉"思考中"状态

- 思考是瞬时的，无法准确检测
- 用"处理中..."作为快速执行时的通用状态
- 避免状态频繁闪烁

### 2. 基于时间阈值显示

- 只有超过阈值才显示具体动作
- 阈值根据动作类型调整：
  - Bash 命令：2s（命令可能卡住）
  - 文件操作：2s（大文件可能慢）
  - 等待模型：5s（正常响应通常 <5s）
  - 等待子代理：3s（子代理任务通常较长）

### 3. 显示持续时间

- 让用户知道"卡了多久"
- 超过警戒阈值（如 30s/60s）显示警告样式

### 4. 简化状态类型

| 原状态 | 新状态 |
|--------|--------|
| thinking | 合并到"处理中..." |
| tool_executing | "执行命令"/"读写文件"/"执行工具"（>阈值时） |
| waiting_llm | "等待响应"（>5s 时） |
| waiting_child | "等待 xxx"（>3s 时） |

---

## 总结

### 核心原则

1. **不追求精确**：快速动作无法精确捕获，不如不显示
2. **关注异常**：只显示超过阈值的"卡顿"状态
3. **提供上下文**：显示等待对象、持续时间
4. **减少闪烁**：用"处理中..."作为过渡状态
